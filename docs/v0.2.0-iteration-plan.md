# QMemory v0.2.0 迭代计划

> 基于 v0.1.0 Benchmark (Overall 71.4%) 深度分析制定  
> 目标: Overall ≥82%, longmemeval-s ≥72%, locomo ≥75%  
> 创建: 2026-03-28

---

## v0.1.0 根因分析摘要

对 144 题中 50 个失败 (score<7) 进行根因归类:

| 根因模式 | 失败数 | 占比 | 严重程度 |
|----------|--------|------|----------|
| 多会话时间线重建失败 | 10 | 20% | 🔴 Critical |
| 记忆提取缺失 (事实从未被提取) | 9 | 18% | 🔴 Critical |
| 跨会话链接失败 (后续更新未关联) | 6 | 12% | 🔴 Critical |
| 细节粒度丢失 (高层摘要丢失数值) | 5 | 10% | 🟡 High |
| 综合推理/数据聚合失败 | 5 | 10% | 🟡 High |
| 实体消歧/关系提取失败 | 4 | 8% | 🟡 High |
| 知识更新失败 (旧事实未被替代) | 3 | 6% | 🔴 Critical |
| 检索相关性不足 (信息存在但搜不到) | 3 | 6% | 🟠 Medium |
| 部分提取 (存了一半) | 3 | 6% | 🟠 Medium |
| 时间计算失败 | 2 | 4% | 🟠 Medium |

**三大核心突破方向:**
1. **写入侧: 提取质量** — 解决 18%+6%+10% = 34% 的提取相关失败
2. **读取侧: 跨会话时间线重建** — 解决 20%+12% = 32% 的时间线/链接失败
3. **维护侧: 知识更新与实体消歧** — 解决 6%+8% = 14% 的更新/消歧失败

## 突破方向 1: 提取质量提升 (预期 +8~12%)

### 1.1 对话角色感知提取 ⚡最高优先
**问题**: extractor.py 收到的是纯文本，无法区分 user/assistant 发言，导致:
- 助手的技术反馈从未被提取 (longmemeval-s q_ssa_4: Falcon 50ms→28ms 完全丢失)
- 用户和助手的观点被混淆

**方案**:
- ingest 传递给 extractor 时保留 `[User]:`/`[Assistant]:` 角色标记
- 提取 prompt 增加指令: "Pay attention to BOTH user and assistant. Extract assistant's concrete feedback, technical advice, and evaluations as `advice` or `discussion` category."
- ExtractedMemory 新增 `speaker: user|assistant|both` 字段

**预期影响**: longmemeval-s single-session-assistant +15%, locomo-real single-fact +10%

### 1.2 细节保留提取 (Anti-Summarization)
**问题**: LLM 提取时过度摘要，丢失关键数值/地点/人名:
- "Melanie喜欢camping" 丢失了 beach/mountains/forest 三个地点
- 合作金额 "4.5万/4组" 被摘要为 "有合作关系"
- 薪资信息 "85万→150万" 被忽略

**方案**:
- 提取 prompt 增加强约束: "NEVER summarize away specific numbers, locations, dates. If the original text says '4组共4.5万', extract '4组共4.5万', NOT '有合作'."
- 新增 anti-summarization 示例 (few-shot):
  ```
  BAD:  "用户喜欢运动" (丢失具体运动类型)
  GOOD: "用户周三南山馆打羽毛球，周末爬梧桐山"
  ```
- 对 importance ≥0.7 的记忆自动回溯原文 chunk，补充被丢弃的细节

**预期影响**: locomo multi-turn +8%, longmemeval-s preference +10%

### 1.3 多轮提取覆盖率检查
**问题**: 单次 LLM 调用的提取覆盖率不足，特别是长 chunk:
- 18 轮对话只提取了 4 条记忆 (locomo-real: 宠物/骑马/彩色玻璃完全遗漏)
- 5 条记忆中遗漏了 festival 2022 年信息

**方案**:
- 提取后执行 **chunk-memory alignment check**: 对每个 chunk，检查其关键实体/事件是否被某条 memory 覆盖
- 具体: 从 chunk 中抽取名词短语/数字/日期 (简单 NLP，非 LLM)，与提取结果交叉比对
- 覆盖率 < 60% 时触发 **补充提取**: 针对遗漏实体发起第二次 targeted LLM 调用
- 这是 Phase 4.2 completeness 的增强版，但聚焦于写入时而非后台

**预期影响**: locomo-real +5% (通过减少提取遗漏)

## 突破方向 2: 跨会话时间线重建 (预期 +6~10%)

### 2.1 Session-Aware 时间线锚定
**问题**: 每个 session 的记忆独立存储，无法串联为连贯时间线:
- "3月考虑→7月购买" 需要跨2个session，但系统只检索到片段
- "P7→P8→P2-3" 晋升轨迹散落在5+个session中

**方案**:
- ingest 时自动为同一 user 的记忆建立 **session_sequence**: 递增编号
- session 结束时生成 **session digest**: "本次讨论了 {topics}，涉及 {entities}，时间跨度 {date_range}"
- recall 时检测 multi-session 查询后:
  - 从 session digests 确定相关 sessions
  - 跨多个 session 执行 targeted search
  - 按 session 顺序组装时间线

**预期影响**: locomo multi-turn +10%, longmemeval-s multi-session +15%

### 2.2 时间线事件自动链接
**问题**: timeline_event_store 有事件但无自动链接:
- "2月签合同" 和 "Q1完成第一组" 未被链接
- "6月晋升P8" 和 "9月跳槽字节" 未被标记为因果链

**方案**:
- 写入时间线事件时，自动搜索同 user 近期事件:
  - 同实体 + 时间相近 → 链接为 `follow_up`
  - 同主题 + 结果/状态变化 → 链接为 `outcome`
- recall assembler 在 timeline section 自动展示链: `[2月] 签合同 → [Q1] 完成第一组 → [6月] Q2夏季主题`

**预期影响**: locomo logical-reasoning +8%, longmemeval-s temporal-reasoning +10%

### 2.3 全年事件索引 (Year Digest)
**问题**: "总结2025年人生轨迹" 这类全年查询几乎全部失败
- 常规 top_k=10 搜索无法覆盖一整年的事件

**方案**:
- 定期 (或按需) 生成 **年度事件索引**: 按月/季度聚合 user 的所有时间线事件
- recall detect "总结/回顾/一年/轨迹" 等关键词时直接查询年度索引
- 不依赖向量搜索，直接按时间范围 SQL 查询

**预期影响**: longmemeval-s multi-session +8%

## 突破方向 3: 知识更新与实体消歧 (预期 +3~6%)

### 3.1 知识更新强化 (Profile Supersede)
**问题**: 用户跳槽后 profile 仍显示 "腾讯" — supersede 链在 profile 层断裂:
- conflict.py 的 supersede 只处理 memory 粒度，不触发 profile 更新
- profile_updates 的 confidence 覆盖逻辑不够强

**方案**:
- conflict.py 发现 `supersede` 时，如果涉及 profile 相关字段 (company/title/location)，
  **主动触发 profile_store update** 而非仅标记 memory relation
- profile_store 增加 `updated_at` 追踪，同 key 的新值始终覆盖旧值
- recall assembler 中 profile section 标注 "最后更新: {date}"

**预期影响**: longmemeval-s knowledge-update +15%

### 3.2 实体消歧增强
**问题**: "哪个张伟要订婚" 混淆了两个同名不同人:
- entity linking 只看名字匹配，不考虑上下文区分

**方案**:
- entity_store 增加 **disambiguation_key**: `{name}_{department}` 或 `{name}_{relation}`
- 写入时: "市场部张伟" 和 "数据分析张伟" 存为不同 entity
- 提取 prompt 增加 few-shot: "当存在同名不同人时，在 text 中保留区分信息 (部门/关系/特征)"
- recall 时: 检测到人名查询后，自动附加 entity binding 提供消歧信息

**预期影响**: qmemory-chinese name-disambig +15%

## Bench 基础设施修复

### B.1 数据集间隔离强化
**问题**: q_ms_5 context 出现其他数据集(Caroline)的记忆 — DELETE 静默失败导致污染

**方案**:
- runner.py: DELETE 后增加 **验证查询**: `GET /v1/memories/?user_id={eval_user}&page_size=1`
  - 如果 total > 0，重试 DELETE 或报错
- 改为每个数据集使用独立 user_id: `eval_{uuid}_{dataset_name}`

### B.2 ReadTimeout 处理
**问题**: `Search failed for q_lr_05: ReadTimeout` — 复杂查询偶尔超时

**方案**:
- bench runner 超时后自动重试一次 (current: 静默跳过得0分)
- server 端对 search 加超时保护，超时返回已有结果
